## 摘要
随着用于代码的大型语言模型（CodeLMs）和开源贡献的激增，未经授权使用源代码数据集的问题日益受到关注。水印作为一种可行的保护机制，可以通过嵌入所有权信号来防止滥用，但现有方法依赖可检测的触发-目标模式，且仅限于源代码任务，忽视了诸如反编译任务等其他场景。本文提出了 DuCodeMark，一种隐蔽且鲁棒的双重用途代码数据集水印方法，能够在源代码任务和反编译任务间通用。DuCodeMark 将每个代码样本解析为抽象语法树（AST），应用特定语言的风格变换以构建隐蔽的触发-目标对，并在部分返回类型样本中注入可抑制的投毒特征，以增强其在水印移除或规避下的鲁棒性。这些特征在正常训练过程中保持不活跃，但在水印被移除时被激活，从而降低模型性能。为验证水印，DuCodeMark 使用基于独立样本 $t$ 检验的黑盒方法。我们在包含 2 个代码任务、2 种编程语言、3 个 CodeLMs 和 6 个解码温度的 72 个设置中对 DuCodeMark 进行了全面评估。结果表明，该方法始终实现了强可验证性（$p < 0.05$）、高隐蔽性（可疑率 $\leq 0.36$）、对水印及投毒攻击的鲁棒性（召回率 $\leq 0.57$），并在水印被移除时导致显著的模型性能下降（Pass@1 降低 28.6%），凸显了其实用性与稳健性。

## Abstract
The proliferation of large language models for code (CodeLMs) and open-source contributions has heightened concerns over unauthorized use of source code datasets. While watermarking provides a viable protection mechanism by embedding ownership signals, existing methods rely on detectable trigger-target patterns and are limited to source-code tasks, overlooking other scenarios such as decompilation tasks. In this paper, we propose DuCodeMark, a stealthy and robust dual-purpose watermarking method for code datasets that generalizes across both source-code tasks and decompilation tasks. DuCodeMark parses each code sample into an abstract syntax tree (AST), applies language-specific style transformations to construct stealthy trigger-target pairs, and injects repressible poisoned features into a subset of return-typed samples to enhance robustness against watermark removal or evasion. These features remain inactive during normal training but are activated upon watermark removal, degrading model performance. For verification, DuCodeMark employs a black-box method based on the independent-samples $t$-test. We conduct a comprehensive evaluation of DuCodeMark across 72 settings spanning two code tasks, two programming languages, three CodeLMs, and six decoding temperatures. The results demonstrate that it consistently achieves strong verifiability ($p < 0.05$), high stealthiness (suspicious rate $\leq$ 0.36), robustness against both watermark and poisoning attacks (recall $\leq$ 0.57), and a substantial drop in model performance upon watermark removal (Pass@1 drops by 28.6%), underscoring its practicality and resilience.

## 一、研究背景与核心问题

- **研究动机**  
  随着代码大模型（CodeLMs）如 GitHub Copilot、StarCoder、DeepSeek-Coder 的普及，代码数据集的未经授权使用风险激增。  
  代码数据集的采集、清洗、去重及授权成本极高，但当前缺乏有效的版权保护和追溯手段。一旦数据集被泄露或被用于未经授权的模型训练，难以追踪来源或证明所有权。

- **现有问题**  
  以往的代码数据集水印方法（如 CodeMark、CoProtector）存在两大缺陷：
  1. **任务局限性**：仅支持源码级任务（如代码补全、搜索、摘要），无法适用于反编译（Decompilation）等结构转化任务。
  2. **鲁棒性不足**：依赖显式触发-目标（Trigger–Target）映射，易被统计检测（如 DeCoMa 技术）识别和清除。

- **核心目标**  
  设计一种能够跨源码与反编译任务通用的、不可察觉且可验证的**水印机制**，同时具备防御水印移除与数据集盗用的双重功能。

## 二、方法论与核心技术设计

### 1. 总体思想

DuCodeMark 提出一种**“双重用途”（Dual-Purpose）**的代码数据集水印方案，将：
- **风格感知水印（Style-aware watermarking）**
- 与 **可抑制中毒机制（Repressible poison injection）**  
  两者结合，实现兼顾**源代码任务与反编译任务**的统一保护框架。

### 2. 整体流程

方法分为三个主要阶段（见 Fig.1）：

1. **候选样本识别**  
   - 使用 Tree-sitter 解析 AST，从数据集中选出适合嵌入的函数。  
   - 规则：  
     - 含有输出相关字符串（如 `printf`、`System.out.print`）。  
     - 含有特定返回类型（如 float/double）。

2. **风格感知水印 & 抑制性投毒注入**  
   - **风格水印设计（W1/W2）**  
     - *W1*：改变命名风格（在 C 中使用 camelCase，在 Java 中使用 snake_case）。  
     - *W2*：为变量/函数名添加类型后缀（如 `_str`, `_f`, `flag`）。  
     - 同时嵌入特殊字符串触发器（如 `"\n\n"`）。  
     - 确保语义与编译结果不变（相似度≥0.97，编译率=100%）。
   - **可抑制性投毒设计（P）**  
     - 针对返回类型 float/double 的函数实施轻量投毒。
     - 例如：  
       - C 中将 `return` 值改为固定常数 `-1.0`；  
       - 引入 typedef/包装类替代类型以迷惑模型。  
     - 投毒样本比控制在候选集的 50%，并与水印关联形成“抑制条件”。

3. **水印验证（黑盒检测）**  
   - 无需访问模型内部，仅通过黑盒 API 调用。  
   - 使用**独立样本 t 检验（p < 0.05）**，比较带触发与不带触发的样本结果。  
   - 当统计显著性达阈值，说明目标模型训练中包含了受保护数据。

## 五、主要结果与发现

1. **有效性**：  
   - 所有水印模型均得到极显著统计差异（p < 1e-5）。  
   - 可在黑盒条件下高可信地判断模型是否使用了带水印数据。

2. **性能无损性**（Harmlessness）：  
   - 在无攻击场景下，Pass@1平均下降仅 0.6%。  

3. **隐蔽性**（Stealthiness）：  
   - 平均可疑率 ≤ 0.36（人类审查几乎无法识别）。  
   - 代码结构复杂度、NLOC、CodeBLEU 均几乎不变。

4. **鲁棒性**：  
   - 抗攻击：对 DeCoMa、KillBadCode、AC、SS 等攻击仍能保持 p < 0.05。  
   - 抗稀释：水印率降至 50% 仍可检测。  
   - 抗编译优化与平台差异：GCC / Clang / O1–O3 下均可验证。

5. **惩罚型防御机制（Punitive Defense）**：  
   - 当攻击者移除水印时，潜在投毒激活，模型性能显著下降（Pass@1 ↓ 28.6%）。

- **抑制性中毒机制**：创新性地将“可逆投毒”与水印绑定，实现水印移除即惩罚。  
- **黑盒验证机制**：无需访问模型参数即可验证模型是否使用受保护数据集。  
- **系统性实验**：涵盖模型、语言、任务、攻击和参数多维评估。  
- **实际适用性强**：兼顾高隐蔽性（低可疑率）与低性能损耗。
